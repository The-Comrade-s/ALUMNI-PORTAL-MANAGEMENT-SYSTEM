# Supabase — Database Setup

## 1. Create the project

1. [supabase.com](https://supabase.com) → New Project.
2. Pick a strong database password — you'll need it for `DB_PASSWORD`.
3. Pick a region close to your Koyeb backend's region to minimize latency
   between the app and the database.

## 2. Get your connection details

Project → **Settings → Database**. Supabase gives you two connection modes;
this app uses both, for different purposes:

| Use | Port | Host pattern | Why |
|---|---|---|---|
| Migrations/seeders (one-off, run during deploy) | `5432` (direct) | `db.<ref>.supabase.co` | Supports prepared statements and DDL without pooler quirks. |
| Normal app traffic (every HTTP request) | `6543` (transaction pooler) | `<region>.pooler.supabase.com` | Supabase's free/small tiers cap direct connections in the low tens — a web app opening one per request exhausts that fast. The pooler multiplexes connections. |

Set your Koyeb environment variables accordingly:

```env
DB_CONNECTION=pgsql
DB_HOST=<pooler-host-from-supabase>       # e.g. aws-0-us-east-1.pooler.supabase.com
DB_PORT=6543
DB_DATABASE=postgres
DB_USERNAME=postgres.<project-ref>         # pooler usernames are prefixed with the project ref
DB_PASSWORD=<your-db-password>
DB_SSLMODE=require
```

For the one-off migration step specifically (see `KOYEB.md`), use the
**direct** connection (port `5432`, host `db.<ref>.supabase.co`,
username `postgres`) instead — either as a one-time override when you run
`php artisan migrate --force`, or by running migrations from your local
machine against the direct connection before deploying.

> Laravel's default `config/database.php` (generated by the Docker build's
> skeleton stage — see `backend/Dockerfile`) already reads
> `sslmode` from `DB_SSLMODE` on the `pgsql` connection. Supabase rejects
> unencrypted connections, so this must be `require`.

## 3. Run migrations

From your local machine (with the backend's `.env` pointed at the
**direct** connection):

```bash
cd backend
php artisan migrate --seed
```

Or, if you'd rather not expose the direct connection locally, run it as a
one-off Koyeb job/shell against the deployed container — see `KOYEB.md`'s
"Running migrations" section.

This creates all 7 migration files' worth of schema (roles, users, the
School→Department→Programme→Graduation Year hierarchy, alumni profiles,
jobs, events, news, messaging, notifications, audit logs, settings) and
seeds: 5 roles + permissions, one super admin account, and a starter
academic structure.

## 4. File storage (Supabase Storage)

Koyeb's container filesystem is ephemeral — anything written to local disk
(profile photos, event gallery images, generated reports if you were to
cache them) is lost on every redeploy or restart. Use Supabase Storage,
which is S3-compatible:

1. Project → **Storage** → New bucket, e.g. `alumni-portal-uploads`. Keep
   it private unless you specifically want public read access to uploaded
   images.
2. Project → **Settings → Storage** → note the S3-compatible endpoint,
   access key, and secret key.
3. Set in your backend environment:
   ```env
   FILESYSTEM_DISK=s3
   AWS_ACCESS_KEY_ID=<from Supabase Storage settings>
   AWS_SECRET_ACCESS_KEY=<from Supabase Storage settings>
   AWS_DEFAULT_REGION=us-east-1
   AWS_BUCKET=alumni-portal-uploads
   AWS_ENDPOINT=https://<project-ref>.supabase.co/storage/v1/s3
   AWS_USE_PATH_STYLE_ENDPOINT=true
   ```

This app's current file-upload wiring is limited — see the root
`README.md`'s scope notes. The storage disk config above is what any
upload code should target once written.

## 5. Row Level Security (RLS)

Supabase enables RLS by default on new tables created through its
dashboard/SQL editor, but **tables created via Laravel migrations are
plain Postgres tables with RLS off** unless you turn it on explicitly.
Since this app's Laravel API is the only thing talking to the database
(with the Postgres user's full credentials, not Supabase's public
anon/service-role split), RLS policies aren't required for this
architecture to be secure — access control happens in Laravel (Policies,
`role:` middleware), not at the Postgres row level. Leave RLS off for
these tables unless you later expose the database directly to
Supabase's auto-generated REST/GraphQL API or client-side Supabase SDK,
in which case you'd need RLS policies mirroring the Laravel Policies.

## 6. Verifying the connection

```bash
psql "postgresql://postgres:<password>@db.<project-ref>.supabase.co:5432/postgres?sslmode=require"
```

If that connects, Laravel will too with the equivalent `DB_*` env vars.

## Honest note on what's verified here

This guide is written from Supabase's documented connection patterns and
standard Postgres/Laravel behavior — it has not been executed against a
live Supabase project in this environment (no network access here). Test
the actual connection string Supabase gives your specific project before
trusting it in production; exact pooler hostnames/ports occasionally
change between Supabase's plan tiers.
